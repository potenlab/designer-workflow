design — let designers & PMs build apps

An auto-triggering Claude Code skill that turns plain-language intent into a working application — no command to type.

Status: Proposal  ·  Owner: dev@potenlab.dev  ·  Date: 2026-06-12  ·  Review: DionNam, Raka  ·  Timeline: 3 days

The goal. A designer or PM describes an application they want — in design and business language — and Claude creates the whole thing. It thinks as an engineer underneath (data, backend, UI, security) but responds as a designer/PM: plain words and a running app, never code. It switches itself on from the user's words — no command to type.

1 · Why

Today, building even a small internal app needs an engineer's mental model — file trees, diffs, databases, RLS. Designers and PMs have the ideas but not that model. The design skill is a translation skin over tools we already have: the user just describes what they want, the skill recognizes the intent and turns itself on, runs the engineer's loop silently, and shows only the result.

6 · The guardrail is the product

The skill builds full apps (data + backend included), so safety isn't "never touch the backend" — it's isolation. Each app is built in its own sandbox and can never reach our live systems. Because the skill fires without an explicit command, this boundary has to hold on its own.

✓ Allowed

  • Everything inside the new app's own workspace/branch
  • Its data model, backend, UI, assets, config

✕ Hard refuse

  • Other apps / repos, production databases
  • Shared secrets, existing auth & billing

If an idea genuinely needs a forbidden change, the skill stops and writes a dev-handoff note for the engineer — in plain language — instead of doing it.

7 · What the user actually experiences

PM: "an intake tool — clients submit a brief, we track it New → Doing → Done" ↓ (no command — the skill recognizes the intent and switches on) design ① restates the plan in plain language "A brief tracker. It saves briefs and their status. It lives in its own sandbox — nothing touches our live data. Go ahead?" ↓ (yes) ② builds the full app on an isolated branch (sandbox only) ③ runs it, shows BEFORE / AFTER — mobile + desktop ↓ (looks good) ④ opens a PR with screenshots + a human summary for the dev

8 · Done when (v1 acceptance)

  • Auto-triggers on app/design intent without a command — and stays quiet on unrelated asks.
  • Builds a runnable app from a plain-language description.
  • Stays in its sandbox — refuses to touch live systems, writes a handoff note instead.
  • Always restates the plan in plain language before building.
  • Always shows the running app (mobile + desktop) before the PR.
  • Always lands on an isolated branch and opens a PR — never commits to main.
  • Never shows the user raw code unless asked.
